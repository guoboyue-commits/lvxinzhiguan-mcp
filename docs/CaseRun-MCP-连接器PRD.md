> 上游战略：[CaseRun-产品战略与MCP路线.md](./CaseRun-产品战略与MCP路线.md)  
> 相关架构：[CaseRun-现状与能力.md](./CaseRun-现状与能力.md)、[CaseRun-Skill-Workflow架构.md](./CaseRun-Skill-Workflow架构.md)

## 1. 背景

律信智管已经有案件、进度、费用、卷宗、材料、文书、服务、名片等业务模块，也已经有内置 CaseRun 对话入口。

现在的机会不是再做一个“AI 聊天窗口”，而是把 CaseRun 变成律师办案系统的操作入口，并进一步通过 MCP 让先进律师可以在 Codex、豆包、Claude、Cursor 或其他 AI 工作台中安全调用自己的案件系统能力。

代码侧已经具备基础：

- `backend/cmd/caserun-mcp` 已提供 `stdio` 与 HTTP MCP 服务入口。
- `backend/internal/caserun/mcp_manifest.go` 已有工具清单。
- `backend/internal/caserun/mcp_server.go` 已有工具注册、只读执行、写操作拦截。
- `backend/internal/caserun/mcp_resources.go` 已有 `case://{case_id}/summary` 与 `case://{case_id}/evidence` 资源。
- `backend/internal/caserun/pending_action.go` 已有确认卡、过期、确认、取消和写入闭环。

当前缺口是产品化，而不是协议从零实现。

## 2. 产品定位

CaseRun MCP 连接器是律信智管对外开放给律师个人 AI 工具的安全办案连接层。

一句话：

> 律师可以在自己的 AI 工作台里查案件、读材料、检索卷宗、计算费用、拟进度和费用草稿；真正写入前，必须回到律信智管确认。

它不是 Codex 专用插件。Codex 是第一批样板入口，MCP 是开放标准入口。

## 3. 目标

第一版目标：

1. 先进律师能用个人 Token 把律信智管接入外部 AI 工具。
2. 外部 AI 工具能安全读取律师本人有权限的案件上下文。
3. 外部 AI 工具能生成中低风险业务草稿，但不能直接落库。
4. 律信智管继续掌握确认、写入、审计和权限边界。
5. 形成可演示、可试点、可复用的先进市场标杆能力。

## 4. 非目标

第一版不做：

- 不做全律所共享 MCP Token。
- 不做外部 AI 直接写数据库。
- 不做高风险删除类动作的外部发起。
- 不做 Codex 专用包装。
- 不做 OAuth 市场化分发。
- 不做面向所有普通律师的复杂配置页。
- 不复制一套独立于 CaseRun 的业务逻辑。

## 5. 用户

### 5.1 第一批用户

愿意主动使用 AI 工具的办案律师。

典型特征：

- 已经在用 Codex、豆包、Claude、Cursor 或类似工具。
- 手里有真实在办案件。
- 愿意用 7 天反馈价值。
- 能接受“AI 生成草稿，系统内确认写入”的责任边界。

### 5.2 暂不优先服务的用户

- 只会使用微信小程序、不愿配置外部 AI 工具的普通律师。
- 需要律所级集中管控、统一分发 Token 的组织客户。
- 期待 AI 自动替律师提交、删除、发布或修改关键业务数据的用户。

## 6. 核心原则

1. **只读优先**：外部 AI 首先应能查、读、搜、算。
2. **写入确认**：任何会改变业务数据的动作，必须生成确认草稿或引导回系统。
3. **个人身份**：第一版使用律师个人 MCP Token，权限等同该律师本人。
4. **系统闭环**：确认、写入、撤销、过期、审计都在律信智管内完成。
5. **复用底座**：MCP 复用 CaseRun Tool、Skill、Workflow、pending_action、write_exec。
6. **可解释**：返回内容必须说明来自哪个案件、哪类材料、哪次查询。
7. **可撤销**：律师可以随时撤销个人 Token。

## 7. 第一版范围

### 7.1 接入方式

默认支持 HTTP MCP。

律师在律信智管中进入“连接 AI 工具”，看到：

- MCP 服务地址。
- 个人 Token 的生成、复制、撤销。
- Codex 等工具的配置示例。
- 最近连接时间。
- 连接测试结果。

`stdio` 仅用于开发、演示或内部测试，不作为普通律师默认接入方式。

### 7.2 只读工具

第一版必须稳定支持：

| 能力 | MCP Tool | 说明 |
|---|---|---|
| 查询案件列表 | `case.list` | 仅返回本人有权案件 |
| 查询案件详情 | `case.detail` | 包含基础信息和当前阶段 |
| 查询进度 | `case.progress.query` | 支持按案件查询 |
| 查询费用 | `case.fee.list` | 支持收支、金额、状态 |
| 查询客户消息 | `case.messages.query` | 注意隐私与范围 |
| 查询附件列表 | `case.file.list` | 返回文件名、类型、时间 |
| 读取材料正文 | `case.evidence.read` | 使用已抽取文本或 OCR 摘要 |
| 检索材料 | `case.evidence.search` | 返回命中的摘录和来源 |
| 查询案件字典 | `case.meta` | 类型、阶段等 |
| 查询导出记录 | `case.export.list` | 历史卷宗/PDF 导出 |
| 查询文书模板 | `legal_doc.list` | 常用模板列表 |
| 计算诉讼费 | `tool.litigation_fee` | 只读计算 |
| 计算利息 | `tool.interest` | 只读计算 |
| 计算违约金 | `tool.penalty` | 只读计算 |
| 计算日期期限 | `tool.date_calc` | 只读计算 |
| 查询节假日 | `tool.holiday` | 只读查询 |

### 7.3 确认草稿

外部 AI 允许为中低风险动作生成确认草稿：

| 动作 | 第一版处理 |
|---|---|
| 新增案件进度 | 生成 `pending_action` |
| 更新案件进度 | 生成 `pending_action` |
| 新增案件费用 | 生成 `pending_action` |
| 更新案件费用 | 生成 `pending_action` |
| 根据材料解析进度草稿 | 生成 `pending_action` 或返回草稿 |
| 导出案件卷宗 PDF | 生成 `pending_action` |
| 生成法律文书 | 生成 `pending_action` |

返回给外部 AI 的结果必须包含：

- `pending_id`
- `preview`
- `risk_level`
- `expires_at`
- `confirm_url`
- 简短说明：需要律师在律信智管内确认后才会执行。

### 7.4 高风险引导

以下动作第一版不允许外部 AI 发起 pending：

| 动作 | 处理方式 |
|---|---|
| 删除案件 | 返回高风险提示，引导回律信智管 |
| 删除进度 | 返回高风险提示，引导回律信智管 |
| 删除费用 | 返回高风险提示，引导回律信智管 |
| 批量修改案件 | 返回高风险提示 |
| 发布、下架或大改服务产品 | 第一版只读或引导 |
| 修改对外律师名片核心信息 | 第一版只读或引导 |

## 8. 功能需求

### FR1：个人 MCP Token

律师可以在律信智管中生成个人 MCP Token。

要求：

- Token 只展示一次明文。
- 系统只保存哈希。
- Token 记录创建时间、最近使用时间、状态。
- 支持撤销。
- 支持重新生成。
- 支持显示 Token 前缀用于识别。

### FR2：MCP HTTP 鉴权

HTTP MCP 接口支持 `Authorization: Bearer <token>`。

要求：

- 兼容现有律师 JWT，便于开发和内部测试。
- 新增个人 MCP Token 鉴权，作为对外推荐方式。
- 鉴权后映射到律师个人 `user_id`。
- 所有工具执行都使用该用户权限上下文。

### FR3：连接设置页

在 CaseRun 或律师个人设置中增加“连接 AI 工具”入口。

页面展示：

- 当前连接状态。
- MCP 服务地址。
- Token 管理。
- 配置示例。
- 连接测试按钮。
- 安全说明。

页面不应让律师理解复杂协议细节，只强调三件事：

1. 复制地址。
2. 复制 Token。
3. 粘贴到 AI 工具配置里。

### FR4：标准返回格式

MCP 工具返回统一结构，便于外部 AI 稳定理解。

只读成功：

```json
{
  "ok": true,
  "tool": "case.detail",
  "summary_text": "案件摘要文本",
  "data": {},
  "sources": [],
  "warnings": [],
  "next_actions": []
}
```

确认草稿：

```json
{
  "ok": true,
  "tool": "case.progress.create",
  "status": "draft_created",
  "pending_id": "uuid",
  "preview": "将为某案新增进度...",
  "risk_level": "medium",
  "expires_at": "2026-06-14T12:00:00+08:00",
  "confirm_url": "https://.../lawyer/caserun?pending_id=uuid"
}
```

高风险拦截：

```json
{
  "ok": false,
  "tool": "case.delete",
  "status": "blocked_high_risk",
  "message": "该动作不能由外部 AI 发起，请回到律信智管内操作。",
  "guidance": "打开案件详情页后由律师本人处理。"
}
```

### FR5：确认链接

当 MCP 生成草稿后，系统返回确认链接。

要求：

- Web CaseRun 打开链接后能定位到对应确认卡。
- 确认卡展示与内置 CaseRun 一致。
- 过期草稿提示重新发起。
- 已取消或已确认草稿提示状态。

### FR6：审计

第一版至少记录：

- Token 创建、撤销。
- MCP 请求时间。
- 用户 ID。
- Tool 名称。
- 是否成功。
- 是否生成 pending。
- 错误码。

不要求记录完整敏感正文，但需要能定位问题。

### FR7：试点数据准备

试点前应为每名律师准备：

- 5-10 个在办案件。
- 每案至少 2-5 条进度。
- 至少若干费用记录。
- 至少若干材料或证据摘录。
- 至少一个可演示的确认草稿场景。

## 9. 权限与安全

权限规则：

- MCP Token 代表律师本人，不代表律所。
- 只能访问该律师在律信智管中已有权限的数据。
- 所有案件查询仍走现有案件权限逻辑。
- 外部 AI 不获得绕过确认写入的能力。

安全规则：

- Token 明文只出现一次。
- Token 可撤销。
- Token 泄露后撤销立即失效。
- 高风险动作不生成 pending。
- 写操作 pending 默认 15 分钟过期，沿用现有机制。

## 10. 验收标准

第一版验收：

1. 律师可在页面生成个人 MCP Token。
2. 使用 HTTP MCP 地址和 Token 可在 Codex 样板环境连接成功。
3. `tools/list` 能看到 CaseRun 工具清单。
4. 查询案件列表、详情、进度、费用、材料检索均能返回真实数据。
5. 读取越权案件失败。
6. 新增进度、新增费用从 MCP 返回确认草稿，不直接落库。
7. 律师打开确认链接后可确认或取消。
8. 确认后数据写入，取消后不写入。
9. 删除类高风险动作被拦截并提示回系统处理。
10. Token 撤销后不能继续访问 MCP。

## 11. 试点成功指标

每名试点律师 7 天内：

- 至少 20 次 MCP 或内置 CaseRun 查询/任务。
- 至少 10 次确认草稿或确认写入。
- 至少 3 次用于真实办案事务。
- 第 7 天愿意继续使用，或愿意接入更多案件。

关键访谈问题：

> 如果下周这个能力不能用了，你会不会想念？

## 12. 第一版发布口径

推荐表达：

> 律信智管让律师可以在自己的 AI 工具里安全调用案件系统。AI 可以帮你查案件、读材料、检索卷宗、拟进度和费用草稿；真正写入前，仍由律信智管确认和审计。

不推荐表达：

> 我们做了 Codex 插件。

> AI 可以自动替律师办案。

> MCP 可以直接改案件系统。
